Hook 通知逻辑链路

这是当前方案的可视化评审页：Claude/Codex 主路径只走官方 hook，hook 只负责把状态转换成标准事件， 当前后端 MVP 统一发系统气泡通知；完整 Tauri 托盘 UI、Toast 点击回调和 PID/标题唤窗 fallback 仍是后续计划。

Windows first Hook 不直接发 Toast 统一 JSON 事件 后台离线直接丢弃 不自动批准权限
当前实现：agent-notify-tray 是 Axum localhost 后台，已支持 Bearer 鉴权的 /events、/sessions、/focus/{sessionId}、Windows Toast 展示、内存 session 表、30 秒去重和 Hook Manager 安装/修复。未实现：完整 Tauri 托盘壳、Toast deep link、activation nonce、PID/进程树/窗口标题 fallback、session 详情页和 agentrun。

    运行时文件

    路径 用途
    scripts/hooks/agent-notify-hook.ps1 本项目实现的通用 hook 入口，负责转换事件。
    scripts/hooks/claude-hook.template.json Claude hook 合并参考模板；当前 Hook Manager 生成等价配置并展开绝对路径。
    %LOCALAPPDATA%\AgentNotify\config.json 本地后台地址、端口、通知和 hook 配置；token 存在单独文件。
    scripts/hooks/codex-hook.template.json Codex hook 合并参考模板；当前 Hook Manager 生成等价配置并展开绝对路径。
    %LOCALAPPDATA%\AgentNotify\hooks\agent-notify-hook.ps1 运行时安装副本，Claude/Codex 用户配置实际调用它。
    %LOCALAPPDATA%\AgentNotify\backups\ Hook Manager 修改用户配置前的自动备份目录。
    %LOCALAPPDATA%\AgentNotify\logs\hook.log 只记录 hook 自身错误，不写完整终端输出。

    事件到通知的映射

    eventType 含义 通知 后台状态
    task.completed 任务完成或进程正常退出。 弹通知 completed
    task.failed 进程异常退出或 hook 映射失败。 弹通知 failed
    user.confirmation_required 需要批准命令、权限或工具调用。 高优先级 waiting_user
    user.input_required 需要用户继续输入下一步。 高优先级 waiting_user
    tool.blocked 权限、沙箱、网络或工具调用被阻塞。 弹通知 waiting_user 或 failed
    heartbeat 会话仍存活。 不弹 running

    通知样式设计

    需要确认

    标题：Claude/Codex 需要确认

    正文：项目名 · 会话名 · 点击返回终端

    详情：只显示脱敏摘要，例如 Bash 请求执行命令，参数已隐藏。

    等待输入

    标题：Claude/Codex 等待输入

    正文：项目名 · 当前会话 · 需要你的下一步

    详情：默认不显示模型完整问题。

    完成或失败

    标题：任务完成 / 执行受阻

    正文：项目名 · 当前会话 · 点击查看结果

    详情：只显示退出码或失败类别。

    当前 Windows MVP 只承诺展示 Toast 文本；点击通知主体打开窗口、忽略和静音此项目均待 Tauri/Toast 集成，不在通知里批准命令。

    离线处理

    hook 生成事件 -> 尝试发送后台 -> 后台不可用 -> 直接丢弃
    • hook 默认 2 秒内结束。
    • 后台不可用时默认不重试。
    • 失败不阻塞 Claude/Codex 主流程。
    • hook 默认退出码为 0。
    • 不写离线队列，不补发过期通知。
    离线事件全部丢弃，避免后台恢复后突然弹出一批过期通知。

    隐私默认值

    通知只显示工具名、项目名、事件类型和简短状态。

    • 不显示完整命令
    • 不显示完整终端输出
    • message.detail 最多 160 字符
    • 默认不持久化 raw payload

    禁止动作

    hook 只上报状态，不代表用户做决定。

    • 不自动批准权限请求
    • 不执行额外命令
    • 不上传 payload

    唤窗策略

    当前通过 HTTP focus 按 HWND 定位窗口；点击通知和多级 fallback 待实现。

    • HWND
    • PID 待实现
    • 唯一窗口标题待实现

    监听开关

    当前监听状态只在启动时读取配置；应用内监听开关待 Tauri UI 实现。

    • 监听开启：接收 hook 事件
    • 监听关闭：事件直接丢弃
    • 后台离线：不补发通知

    需要你确认的判断点

    • Hook 是否只负责上报，不直接调用 Windows Toast？

      当前逻辑把通知能力集中在 agent-notify-tray 后台，方便统一去重和安全策略；点击回调待实现。

    • Codex 是否优先使用官方 lifecycle hooks？

      当前逻辑使用 PermissionRequest、Stop、PostToolUse 映射到统一通知事件。

    • 事件入口是否接受 agent-notify emit --stdin 和带 token 的本地 HTTP？

      CLI 入口更适合 hook 调用；HTTP 入口更方便调试，但必须带本地 token。

    • 后台离线或监听关闭时，是否直接丢弃事件？

      当前逻辑不写离线队列，不补发过期通知，避免恢复后刷屏。

    • 通知默认是否隐藏完整命令和敏感 payload？

      当前逻辑只展示简短摘要，完整处理仍回到终端窗口完成。
